# Chapter 6: Memory Management

## 6.1 Introduction

Memory management is one of the most critical functions of an operating system. It determines how physical memory (RAM) is allocated to processes, how processes are protected from each other, and how we can run programs larger than available physical memory.

The goals of memory management include efficient allocation of memory to processes, protection between processes so one can't corrupt another's memory, support for controlled sharing of memory between processes, logical organization that supports program structure like code, data, and stack segments, and physical organization that manages the memory hierarchy including RAM and disk.

When you write `int x = 5;` in C++, the variable `x` needs a memory address. This binding between variable names and addresses can happen at different times. Compile-time binding is when the address is known at compile time, which is rare and mostly used in embedded systems. Load-time binding is when the address is determined when the program is loaded into memory. Execution-time binding is when the address can change during execution, which requires hardware support. Modern systems use execution-time binding with hardware support from the Memory Management Unit.

## 6.2 Logical vs Physical Address

Understanding the distinction between logical and physical addresses is fundamental to understanding memory management.

A **logical address**, also called a virtual address, is the address generated by the CPU. This is what your program sees. When your C++ code accesses a variable, it uses a logical address. A **physical address** is the actual address in RAM — the real location in the memory chips.

The **Memory Management Unit (MMU)** translates logical addresses to physical addresses at runtime. This translation happens on every memory access, so it must be extremely fast. The CPU generates a logical address, the MMU translates it to a physical address, and then the physical address is sent to the memory bus to access RAM.

This separation is powerful. Each process thinks it has its own contiguous memory starting from address 0. The OS and MMU work together to map these virtual addresses to actual physical locations. This provides isolation — one process can't corrupt another's memory because they have different mappings. It also provides flexibility — physical memory can be fragmented while logical memory appears contiguous.

## 6.3 Contiguous Memory Allocation

The simplest approach to memory allocation gives each process a single contiguous block of physical memory.

In **fixed partitioning**, memory is divided into fixed-size partitions at boot time. Each partition can hold one process. The problem is **internal fragmentation** — if a process is smaller than its partition, the remaining space is wasted inside the partition. Also, the number of partitions limits how many processes can run concurrently.

In **variable partitioning**, partitions are created dynamically based on process size. When a process arrives, we allocate exactly as much memory as it needs. The problem is **external fragmentation** — as processes come and go, free memory becomes scattered in non-contiguous holes. The total free memory might be enough for a new process, but no single hole is big enough.

When allocating memory, we need to choose which hole to use. **First Fit** uses the first hole that's big enough — it's fast and generally performs well. **Best Fit** uses the smallest hole that's big enough — it minimizes leftover space but is slow because it must search all holes, and it tends to leave many tiny unusable fragments. **Worst Fit** uses the largest hole — the idea is to leave larger leftovers, but it performs poorly in practice. First Fit is generally preferred.

**Compaction** can fix external fragmentation by moving all processes together to create one large free block. However, compaction is expensive because it requires moving processes and updating all their addresses.

## 6.4 Paging

**Paging** divides both physical memory and logical memory into fixed-size blocks, eliminating external fragmentation entirely.

A **page** is a fixed-size block of logical memory, typically 4KB. A **frame** is a fixed-size block of physical memory, the same size as a page. The **page table** maps pages to frames for each process. The key insight is that pages don't need to be contiguous in physical memory — any free frame can hold any page.

For address translation, a logical address is divided into two parts: the page number and the page offset. The page number is used as an index into the page table to find the corresponding frame number. The physical address is then the frame number concatenated with the offset.

For example, with 4KB pages (4096 bytes = 2^12), the offset is 12 bits. If a logical address is 8196, the page number is 8196 / 4096 = 2, and the offset is 8196 % 4096 = 4. If the page table says page 2 maps to frame 7, the physical address is 7 × 4096 + 4 = 28676.

Each page table entry contains more than just the frame number. The **valid/invalid bit** indicates whether this page is currently in memory. **Protection bits** specify read, write, and execute permissions. The **dirty bit** indicates whether the page has been modified. The **reference bit** indicates whether the page has been accessed recently.

The **Translation Lookaside Buffer (TLB)** is a critical optimization. Since the page table is in memory, every memory access would require two memory accesses — one to the page table and one to the actual data. The TLB is a fast hardware cache for page table entries. On a TLB hit, translation is immediate. On a TLB miss, we access the page table and update the TLB. With a typical hit ratio of 98-99%, the overhead of paging is minimal.

A significant challenge is page table size. For a 32-bit address space with 4KB pages, we have 2^32 / 2^12 = 2^20 = 1 million pages. If each entry is 4 bytes, the page table is 4MB per process! Solutions include hierarchical (multi-level) page tables that only allocate entries for used portions of the address space, hashed page tables good for sparse address spaces, and inverted page tables with one entry per physical frame rather than per page.

## 6.5 Segmentation

**Segmentation** divides memory according to the logical structure of a program: code segment, data segment, stack segment, heap segment. Each segment can be of different size, grow independently, and have different protections.

A segment table maps segment numbers to physical locations using a base address and a limit. A logical address is a pair of (segment number, offset). Translation checks if the offset is within the limit, then adds the offset to the base address.

Segmentation matches how programmers think about programs — code, data, and stack are separate logical units. It makes sharing and protection natural: share the code segment between processes, make it read-only. But variable-size segments cause external fragmentation.

Modern systems combine segmentation with paging: segments for logical organization, paging within segments to eliminate fragmentation. The Intel x86 architecture supports both, though modern operating systems mostly use a flat memory model with paging.

## 6.6 Virtual Memory

**Virtual memory** allows processes to use more memory than physically available by using disk as an extension of RAM. Not all of a process's memory needs to be in RAM at once — we keep frequently used parts in RAM and store the rest on disk in the swap space, bringing pages into RAM on demand.

The benefits are substantial. Processes can use more memory than physical RAM. More processes can run simultaneously since each only needs part of its memory in RAM. Program loading is more efficient because we don't load the entire program at startup. Programming is simplified because the programmer doesn't worry about physical memory size.

When I allocate a large array with `new` in C++, I get virtual memory. Physical RAM is allocated only when I actually touch those pages. This is why I can allocate more than physical RAM — as long as I don't use it all at once.

## 6.7 Demand Paging

**Demand paging** is the strategy of loading pages into memory only when they're needed, not in advance. In pure demand paging, a process starts with zero pages in memory, and every initial access causes a page fault.

When a process accesses a page not in RAM, a **page fault** occurs. The CPU generates a trap to the OS. The OS checks if it's a valid address — if not, it's a segmentation fault. The OS finds a free frame, or if none is free, evicts a page using a page replacement algorithm. The OS reads the page from disk into the frame, which takes about 10 milliseconds. The OS updates the page table to mark the page as valid and set the frame number. Finally, the OS restarts the instruction that caused the fault.

The performance impact of page faults is critical to understand. The effective access time formula is: EAT = (1-p) × memory_access_time + p × page_fault_time, where p is the page fault rate. With memory access at 100 nanoseconds and page fault time at 10 milliseconds (10,000,000 nanoseconds), even a 0.1% page fault rate (p = 0.001) gives: EAT = 0.999 × 100 + 0.001 × 10,000,000 = 99.9 + 10,000 = 10,100 nanoseconds. That's 100 times slower than pure memory access! This is why page replacement algorithms are so important.

**Copy-on-Write (COW)** is an optimization for `fork()`. When a process forks, instead of copying all pages, parent and child initially share all pages marked as read-only. When either process writes to a page, a page fault occurs, and the OS creates a copy for the writer. This avoids copying pages that are never modified, making `fork()` much faster.

## 6.8 Page Replacement Algorithms

When a page fault occurs and there's no free frame, we must evict a page. The goal is to minimize page faults by choosing the page least likely to be needed soon.

**FIFO (First-In, First-Out)** replaces the oldest page — the first one loaded. It's simple to implement using a queue, but it ignores usage patterns and may evict frequently used pages just because they were loaded first. FIFO also suffers from **Belady's anomaly**, where adding more frames can sometimes cause more page faults, which is counterintuitive.

**Optimal (OPT)** replaces the page that will not be used for the longest time in the future. This gives the minimum possible page faults and serves as a benchmark. However, it's impossible to implement because we can't know the future. It's useful for comparing other algorithms against the theoretical best.

**LRU (Least Recently Used)** replaces the page that hasn't been used for the longest time, using past behavior to predict future behavior. It's a good approximation of optimal because of locality — if a page was used recently, it's likely to be used again. LRU doesn't suffer from Belady's anomaly. However, exact LRU is expensive to implement because it requires tracking access times for every page.

Practical systems use **LRU approximations**. The **Clock algorithm** (also called Second Chance) uses a reference bit that hardware sets when a page is accessed. Pages are arranged in a circular list. When we need to evict, we check the reference bit: if it's 1, we clear it and give the page a second chance; if it's 0, we evict the page. This approximates LRU with minimal overhead.

## 6.9 Thrashing

**Thrashing** occurs when a system spends more time paging than executing useful work. Pages are constantly swapped in and out, and the CPU utilization drops dramatically.

The cause is when processes don't have enough frames for their **working set** — the set of pages they're actively using. When one process page faults, it steals a frame from another process. That process then page faults and steals a frame back. This cycle continues, and the system spends all its time paging.

The symptoms are high disk I/O, low CPU utilization, and the system feeling frozen. It's a death spiral — the more processes page fault, the more they interfere with each other.

Prevention strategies include the **working set model**, which ensures each process has enough frames for its working set. If the total working set size exceeds available frames, we suspend some processes. The **page fault frequency** approach monitors the fault rate per process — if it's too high, allocate more frames; if it's too low, take some frames away. If no frames are available and fault rate is high, suspend the process. Other solutions include reducing the degree of multiprogramming or simply adding more RAM.

## 6.10 Memory in C++

Understanding how C++ memory relates to OS concepts is valuable.

The **stack** holds local variables and function call information. Allocation is automatic — just move the stack pointer. It's very fast with no system call needed. However, stack size is limited, typically 1-8 MB. Stack overflow causes a segmentation fault.

The **heap** is where dynamic allocation happens with `new` and `delete`. It may involve system calls for large allocations. Memory must be manually managed (or use smart pointers).

The allocation layers are: `new`/`delete` calls the C++ runtime's `operator new`/`delete`, which calls the C library's `malloc`/`free`, which maintains a memory pool and only calls the OS (`brk()` or `mmap()`) when the pool is exhausted. Small allocations come from the pool with no system call overhead. Large allocations typically use `mmap()` to get new pages directly from the OS.

## 6.11 Common Interview Questions

**Q: What's the difference between paging and segmentation?**

Paging divides memory into fixed-size pages — it's a physical view, transparent to the programmer. Segmentation divides memory into variable-size segments based on logical structure — code, data, stack. Paging eliminates external fragmentation but has internal fragmentation. Segmentation matches program structure but has external fragmentation. Modern systems combine both.

**Q: Explain how a page fault is handled.**

When the CPU accesses a page not in memory, hardware traps to the OS. The OS checks if it's a valid address. It finds a free frame or evicts a page. It reads the page from disk into the frame, which takes about 10ms. It updates the page table. It restarts the instruction. The key cost is disk I/O — about 10ms, which is huge compared to 100ns memory access.

**Q: Why is LRU better than FIFO?**

LRU uses past behavior to predict future — if a page was used recently, it's likely to be used again due to temporal locality. FIFO ignores usage patterns and might evict a frequently used page just because it was loaded first. LRU also doesn't suffer from Belady's anomaly where more frames cause more faults.

**Q: What is thrashing and how do you prevent it?**

Thrashing is when the system spends more time paging than executing because processes don't have enough frames for their working sets. Prevention involves ensuring each process has enough frames for its working set. If total demand exceeds available memory, suspend some processes rather than letting everyone thrash. Monitor page fault rates and adjust allocations.

## Key Takeaways

Memory management is about creating abstractions that simplify programming while efficiently using physical resources. Paging solves external fragmentation by dividing memory into fixed-size blocks — any free frame can hold any page. The TLB makes paging practical by caching page table entries with a 98%+ hit rate.

Virtual memory decouples the programmer's view from physical reality. A process can have a large address space even with limited RAM. The OS and hardware create this illusion, bringing pages from disk on demand.

Demand paging is efficient but page faults are expensive — even a 0.1% fault rate causes 100x slowdown. This is why page replacement algorithms matter. LRU is the best practical algorithm, and approximations like the Clock algorithm are used in real systems.

Thrashing is a death spiral where the system spends all its time paging. Prevent it by ensuring working sets fit in memory. Copy-on-write optimizes `fork()` by sharing pages until they're modified.

Understanding virtual memory helps write efficient C++ code. When you allocate a large array, you're getting virtual memory, not physical. Physical allocation happens on first access. This means you can allocate large sparse data structures efficiently — only touched pages consume RAM.
